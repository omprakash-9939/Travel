# Repo Setup

A consistent, conventional repository is what lets humans and agents navigate the same way and what keeps a build from sprawling. This skill scaffolds that structure and — crucially — **seeds the agent's context** so every build session starts grounded instead of guessing.

## Workflow

1. **Confirm the stack.** Ask or infer: React/Node (Next.js full-stack or NestJS API), .NET (ASP.NET Core), or Java (Spring Boot)? **Default to React + Node with Material UI for an easy local POC when nothing is specified** — Vite + React for a simple SPA, or Next.js for full-stack in one app, NestJS when a separate API is wanted. Respect stated preferences (the team also runs Nest/Next; individuals have their own). Pin versions. The layout and tooling follow from this — see `../architecture-grounding/references/stack-options.md`.
2. **Reflect the layers in the layout.** Give domain logic and data access a home that isn't the component or the controller — even a thin one. A typical minimal split: UI / application(API) / domain / data / integrations, plus cross-cutting (auth, config, logging). Include only the layers the POC needs (see the "Layers" section of `stack-options.md`).
3. **Lay out the structure** for that stack (see the per-stack sketches below). Favour a clear separation of app code, tests, and the pipeline artifacts.
4. **Set conventions** — formatter and linter, the test runner and where tests live, an `.editorconfig`, and **commit/PR conventions** written down so humans and the agent follow the same etiquette. These are what `bdd-to-tdd` detects and what keeps the build consistent; make them explicit, not implicit. The commit conventions are simple, standard, and worth stating outright:
   - **Commit small and often** — one logical change per commit, committed as soon as it's green, not batched into one giant end-of-day commit. Frequent commits give you a safety net, clean history, and easy rollback. (Branch first, then commit often.)
   - **Messages say _why_, not just _what_** — the diff already shows what changed; the message explains the reason.
   - **Never commit secrets or `.env`** — keep them in `.gitignore` and ship a `.env.example` instead.
   - **A human owns every commit and merge.** The agent writes; the person who approves is accountable. Keep messages clean — no "Generated by …" / "Co-Authored-By" footers; the author of record is the developer.
5. **Create the pipeline folders** so discovery and specs live with the code:
   ```
   docs/discovery/        # discovery artifacts
   requirements/          # epics/stories/functions (requirements-breakdown)
   architecture/          # grounding + ADRs + build-context.md
   design-system/         # tokens, components, patterns
   features/              # *.feature (requirements-to-bdd)
   <tests dir>            # per the stack's convention
   ```
6. **Write `CLAUDE.md`** at the repo root pointing the agent at the grounding (this is the highest-value step):
   ```
   # Project context for agents
   Before building, read architecture/build-context.md and design-system/README.md.
   Build against the verified features/*.feature and make the tests pass.
   Conventions: <link CONTRIBUTING / the conventions section>.
   ```
7. **Add a README and CONTRIBUTING** capturing the stack, how to run, test, and the conventions. Initialise version control and a `.gitignore` for the stack.

## Per-stack layout sketches (keep it light; adapt)

- **Next.js (full-stack React):** `app/` (routes), `components/`, `lib/`, co-located `*.test.tsx`, `e2e/`, `features/`.
- **React + NestJS:** `apps/web/` (React) and `apps/api/` (Nest modules/controllers/services), shared `packages/`, tests alongside, `features/`.
- **.NET (ASP.NET Core):** `src/<Project>/` (Controllers/Services/Domain/Data), `tests/<Project>.Tests/`, `features/`.
- **Java (Spring Boot):** `src/main/java/...` (controller/service/repository/domain), `src/test/java/...`, `features/`.

## Quality bar

- Stack chosen and **pinned to versions**.
- Conventions (lint/format/test/commit) explicit and documented.
- Pipeline artifact folders present so discovery, specs, and code live together.
- `CLAUDE.md` points the agent at `architecture/build-context.md`, `design-system/`, and `features/`.
- A newcomer (human or agent) can run and test the project from the README alone.

## Anti-patterns

- **Assuming a stack** — the layout and tooling differ; confirm first.
- **Implicit conventions** — "we just know" doesn't survive an agent or a new hire; write them down.
- **No agent context** — skipping `CLAUDE.md` means every build session re-derives the project from scratch and drifts.
- **Premature monorepo/microservices** — for a POC, prefer the simplest structure that fits; don't over-engineer the repo.
